## レビュー観点（PR に付けるコメントの模範）

`docs/PULL_REQUESTS.md` に受講生向けの一覧あり。指導側のチェックリスト:

### 必ず指摘したい
- [ ] キーワード検索は closure でグルーピングされているか（#12 の本質を理解しているか）
- [ ] 削除はフォーム + `@method('DELETE')` になっているか（HTTP メソッドの理解）
- [ ] `unique` は登録・編集の両方で、編集は自分を除外できているか
- [ ] SQLインジェクション対策がバインド/Eloquent になっているか
- [ ] XSS 対策が `{{ }}` になっているか（`{!! !!}` を残していないか）
- [ ] N+1 が 1クエリ集計に変わっているか

### できれば指摘したい
- [ ] マジックナンバーが定数化されているか
- [ ] バリデーションが FormRequest に出ているか
- [ ] メール送信の重複が共通化されているか
- [ ] 変数名が意味のある名前に変わっているか
- [ ] 不要コード・TODO が消えているか

## よくある「直したつもり」ミス（指導ポイント）

- **#12**: `orWhere` を消して `where` だけにしてしまい、カナ・メール検索ができなくなる → closure で括るのが正解。
- **#23**: 登録に `unique` を付けたが編集で自分自身を除外し忘れ、**自分を編集保存できなくなる**。
- **#33 appends**: `appends(request()->all())` にすると `_token` 等も付く場合がある → `request()->query()` が無難。
- **#27**: `strtotime` を噛ませず `date('Y', $member->birthday)` のまま桁だけ直そうとする → 型の問題だと気づけているか確認。
- **#30**: 条件を外しただけで満足しがち → `from`/`bcc` のハードコードにも触れさせる。
- **セキュリティ**: XSS を直す際に `strip_tags` で妥協 → 出力エスケープ（`{{ }}`）が本筋であることを説明。
