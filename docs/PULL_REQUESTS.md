# Pull Request とレビューコメント（サンプル）

実務では、バグを直したら Pull Request を出し、レビューを受けてからマージします。
ここでは **PRの書き方の例** と、**あなたのコードに付きそうなレビューコメント**をまとめています。
自分の PR に置き換えて使ってください。

---

## PR テンプレート

```markdown
## 対応 Issue
Close #12

## 変更内容
- 検索条件の OR グルーピングを修正
- 退会者がデフォルトで表示されないよう修正

## 確認方法
1. 一覧で氏名検索する
2. 退会者が混ざらないこと、ステータス絞り込みが効くことを確認

## 影響範囲
- MemberController@index のみ
```

---

## レビューコメント例（Reviewer 目線）

実際にレビュワーが付けそうなコメントです。自分のコードを見直すチェックリストとしても使えます。

### 設計・可読性

- 💬 「`MemberController` が長すぎます。検索条件の組み立てやメール送信は **Service クラスに切り出せそう**です。」
- 💬 「`$data` `$tmp` `$flg` `$value` など、**変数名から中身が分かりません**。`$member` `$mailData` `$isGold` のように意図が伝わる名前にしましょう。」
- 💬 「`if` のネストが深いです。**早期 return** やマッチ式でフラットにできます。」
- 💬 「ランク判定（`1 / 2 / 3`）や ステータス（`0 / 1 / 2`）が**マジックナンバー**になっています。定数か Enum にしませんか。」
- 💬 「同じメール送信処理が `store` と `sendMail` に**重複**しています。共通化しましょう。」
- 💬 「`noreply@example.com` や `admin@example.com` が**ハードコード**されています。`config` / `.env` に出しましょう。」

### バリデーション

- 💬 「バリデーションが Controller に直書きされています。**FormRequest** に切り出すと再利用できます。」
- 💬 「`store` にはチェックがあるのに `update` には**ほぼ無い**のはなぜでしょう。編集でも同じルールを効かせたいです。」
- 💬 「メールアドレスの**重複チェック（unique）が抜けています**。」

### パフォーマンス

- 💬 「一覧のループ内で毎回 `count()` クエリを投げていて、**N+1（ループ内DBアクセス）**になっています。集計を1クエリにまとめられます。」
- 💬 「`paginate()` のページリンクに検索条件が引き継がれていません。`appends()` が必要です。」

### セキュリティ

- 💬 「`DB::select(\"... like '%\".$request->phone.\"%'\")` は **SQLインジェクション**の危険があります。バインドを使ってください。」
- 💬 「Blade で `{!! $member->memo !!}` を使うと **XSS** の危険があります。`{{ }}` でエスケープしましょう。」

### その他

- 💬 「使われていない `private function formatPhone()` やコメントアウトされた `searchAddress()` は**削除**しましょう。」
- 💬 「`// TODO` が残っています。対応するか Issue 化してください。」

---

## レビュー対応の流れ

1. コメントごとに「対応する / 別Issueにする / 意図を説明する」を返信する
2. 修正したら追加コミットして push
3. すべて解決したら `Resolve conversation`
4. Approve をもらってマージ

> 大事なのは「言われたから直す」ではなく、**なぜそれが問題なのかを説明できる**ことです。
