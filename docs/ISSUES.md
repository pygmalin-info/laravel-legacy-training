## #12 会員検索の結果がおかしい

**症状**
検索ボックスに氏名を入れて検索すると、**退会済みの会員まで一覧に出てくる**ことがある。
また、ステータスで「有効」を選んで氏名検索しても、絞り込みが効いていないように見える。

**再現手順**
1. 会員一覧を開く
2. 適当な氏名キーワードで検索する
3. 退会済み（ステータス=退会）の会員が結果に混ざることを確認

**期待**
検索条件（キーワード・ステータス）が正しく AND で効くこと。退会者はデフォルトで表示しない。

`labels: bug, priority:high`

## #14 一覧の並び替えが正しく動かない

**症状**

- 昇順・降順が切り替えられない（常に昇順のまま）
- 存在しない項目を指定するとエラーになる懸念がある

**期待**
クリックのたびに昇順・降順が切り替わる。並び替え対象は決められた列だけに限定されていること。

`labels: bug`

## #33 一覧が重い／検索してページ送りすると条件が消える

**症状（2件）**
1. 会員数が増えると一覧の表示が遅い。ページを開くだけで大量の SQL が走っている。
2. 検索や並び替えをした状態で 2ページ目に進むと、**検索条件が消えて全件に戻ってしまう**。

**再現手順**
1. `php artisan db:seed` 後に一覧を開き、発行 SQL 数を確認（Debugbar 等）
2. キーワード検索 → 2ページ目のリンクをクリック → 条件が消える

**期待**
一覧表示の SQL 発行数が会員数に比例して増えないこと。ページ送りしても検索条件が維持されること。

`labels: bug, performance`

## #7 【セキュリティ】検索とメモ欄に脆弱性の懸念

**症状**
1. 電話番号検索で、入力値がそのまま SQL に渡っているように見える（**SQLインジェクション**の懸念）。
2. メモ欄に HTML やスクリプトを入れて保存し、詳細画面を開くと**そのまま実行されてしまう**（**XSS**の懸念）。

**再現手順**
1. 電話番号検索欄に `' or '1'='1` のような値を入れて挙動を見る
2. メモに `<script>alert(1)</script>` を保存 → 詳細画面を開く → alert が出る

**期待**
ユーザー入力を安全に扱うこと（プレースホルダ利用／出力エスケープ）。

`labels: security, priority:high`
