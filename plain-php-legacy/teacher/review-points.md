# レビュー観点・レガシーポイント・採点／素PHP版

## レガシーポイント一覧

| 分類 | 場所 | 内容 |
|------|------|------|
| SQL文字列連結 | ほぼ全ページ | ユーザー入力を直接SQLに連結（SQLi） |
| 出力未エスケープ | `index.php` / `member_view.php` | `echo $値` 直出し（XSS） |
| 脆弱なパスワード | `login.php` / `init.sql` | md5 で保存・比較 |
| CSRF対策なし | 全フォーム | トークン検証が無い |
| 認可の粒度 | 全ページ | ログインチェックのみ。権限区分なし |
| 命名が曖昧 | 全体 | `$flg` `$tmp` など |
| if ネスト | `member_add.php` | ランク判定の三重ネスト |
| マジックナンバー | 全体 | status/rank/gender の数値直書き |
| ハードコード | `config.php` ほか | メール宛先など（config化はされている方） |
| 重複コード | `member_add.php` / `mail.php` | メール組み立てが重複 |
| HTML/PHP混在 | 全ページ | ロジックと表示が同居 |
| N+1 | `index.php` | ループ内 COUNT |
| コメント不足 | 全体 | 意図の説明が少ない |

## レビューで必ず指摘したい
- [ ] **全ページ**の SQL がプレースホルダになっているか（1ページでも連結が残れば穴）
- [ ] 出力が `h()` を通っているか（`memo` は `nl2br(h())`）
- [ ] パスワードが `password_hash()`/`password_verify()` になっているか
- [ ] 削除が POST になっているか（GET削除は CSRF で悪用可能）
- [ ] 検索の OR がカッコで括られているか
- [ ] 重複チェック・必須チェックが登録/編集の両方にあるか

## できれば指摘したい
- [ ] CSRF トークンの導入
- [ ] マジックナンバーの定数化（`functions.php` に定数を集約）
- [ ] メール組み立ての共通関数化
- [ ] N+1 の1クエリ集計化、ページャの条件引き継ぎ

## よくある「直したつもり」ミス
- SQLi対策で `mysqli_real_escape_string` だけに頼る → **プレースホルダが本筋**（LIKEやORDER BYは特に危険）。
- XSSを `strip_tags` で妥協 → 出力エスケープ（`h()`）が正解。
- ORDER BY はプレースホルダに**できない**ので、**ホワイトリスト**で対応する点に気づけているか。
- 検索の OR を消して AND だけにしてカナ/メール検索を壊す → カッコで括るのが正解。
- md5→password_hash 移行時、既存データとの互換をどうするか（教材では作り直しでOK、実務では段階移行）。
